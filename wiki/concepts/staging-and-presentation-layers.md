# Staging and Presentation Layers (два слоя DW)

**Источник:** [The Data Warehouse Toolkit](../sources/the-data-warehouse-toolkit.md) (контекст методологии Kimball; терминология в командах может отличаться)

## Суть

**Разделение на два слоя** в типичном контуре витрин — это разделение ответственности между:

1. **Слой приёма (staging / staging area)** — данные **из источников**, максимально близко к тому, «что пришло», плюс техническая обвязка загрузки.
2. **Слой представления (presentation / dimensional layer / витрина)** — **бизнес-модель** для аналитики: факты, измерения, grain, SCD, согласованные справочники.

Пользователи BI и отчёты работают со **вторым** слоем; первый — для инженеров данных, качества и **пересчётов**.

---

## Слой 1: Staging (staging area, raw layer)

**Задача** — **забрать** данные из OLTP, API, файлов, потоков и **сохранить** так, чтобы можно было:

- **повторно** прогнать преобразования в витрину;
- **расследовать** расхождения (что было в источнике vs что попало в звезду);
- **не зависеть** от доступности источника в момент построения фактов.

**Типичные свойства**

| Аспект | Практика |
|--------|----------|
| Структура | Часто **близка к источнику** (та же гранулярность или чуть унифицированная); не обязательно звезда. |
| Ключи | **Natural / source keys**, иногда хэши строк для дедупа; **не** обязательно surrogate для бизнеса. |
| История | Пакеты загрузок (`batch_id`, `loaded_at`), иногда **SCD на staging не делают** — история бизнеса живёт во **втором** слое. |
| Аудитория | ETL/ELT, data quality, поддержка — **не** финальные пользователи отчётов. |

**Staging — не «мусорка»:** в нём фиксируется **контракт** «что мы считаем источником истины на момент загрузки». Без него сложно отвечать на вопросы вида «почему цифра в отчёте изменилась после перезагрузки».

## Слой 2: Presentation (витрина, dimensional warehouse)

**Задача** — выразить **бизнес-процессы** в модели Kimball: [Dimensional Modeling (Kimball)](dimensional-modeling-kimball.md), [Star Schema and Grain](star-schema-and-grain.md), [Dimension Tables](dimension-tables.md), [Slowly Changing Dimensions](slowly-changing-dimensions.md).

**Типичные свойства**

| Аспект | Практика |
|--------|----------|
| Модель | **Fact + dimensions**, grain зафиксирован; при необходимости **snowflake** отдельных узлов. |
| Ключи | **Surrogate keys** в измерениях, согласование **conformed dimensions** между витринами. |
| Семантика | Меры **additive / semi-additive** осмысленны для бизнеса; атрибуты измерений — для срезов и иерархий. |
| Аудитория | BI, аналитики, приложения на SQL поверх витрины. |

Поток: **staging → правила интеграции/очистки → загрузка в факты и измерения**. Иногда «очистку» выделяют в **третий физический** слой (см. ниже) — логически это всё равно **путь из приёма в presentation**.

---

## Зачем именно два логических уровня

| Причина | Объяснение |
|---------|------------|
| **Изоляция от источника** | Схема OLTP меняется — staging подстраивается быстрее; **витрина** эволюционирует по **бизнес-контракту**, а не по каждому релизу приложения. |
| **Идемпотентность и перезапуск** | Ошибку в преобразованиях исправляют, **пересчитав** витрину из staging без повторного выгребания всего из прод-БД. |
| **Lineage** | Понятная цепочка: источник → staging → факт/измерение → дашборд. |
| **Разные SLA** | Staging можно наполнять **часто** (near-real-time append); витрину — по расписанию сложных преобразований. |
| **Единая семантика** | Несколько источников сходятся в **conformed dimensions** только во втором слое ([Conformed Dimensions](conformed-dimensions.md)). |

## Два слоя vs три физических слоя

В реальных проектах часто говорят о **трёх** зонах: **raw / clean / curated**, или **bronze / silver / gold** (medallion). Это **не отменяет** логическое деление:

- **Raw / bronze** ≈ расширенный **staging** (ещё сырее или дольше хранится «как есть»).
- **Silver / clean** ≈ нормализованные, очищенные сущности **между** приёмом и звездой.
- **Gold / curated** ≈ **presentation**, витрины под продукты.

Для собеседования важно уметь сказать: **два слоя** — это **приём (source-aligned)** и **представление (business-aligned)**; третий слой — **уточнение пайплайна**, а не другая теория.

---

## Связь с ETL и batch

Загрузка витрин обычно **пакетная** ([Batch Processing](batch-processing.md)): extract в staging, transform в dimensional слой, publish для BI. В ELT схема похожа: **load raw** (staging), **transform в DWH** (presentation) в SQL/движке.

## Типичные вопросы на интервью

**Q: Зачем нужен staging layer, если можно сразу трансформировать в витрину?**
A: 1) **Идемпотентность**: при ошибке в трансформации — пересчитываешь из staging, не перезагружая из прод-БД. 2) **Аудит**: можно проверить, что пришло от источника vs что попало в витрину. 3) **Развязка**: источник может быть недоступен в момент rebuild. 4) **Разные SLA**: staging пополняется часто, витрина — по расписанию сложных трансформаций.

**Q: Чем bronze/silver/gold отличается от staging/presentation?**
A: Это расширение той же идеи. Bronze ≈ raw staging (как пришло). Silver ≈ cleaned/normalized (между приёмом и витриной). Gold ≈ presentation (бизнес-витрины для BI). Два слоя Kimball — это **логическое** деление (source-aligned vs business-aligned); medallion — **уточнение** с промежуточной зоной.

**Q: Кто должен иметь доступ к staging, а кто к presentation?**
A: Staging — для **инженеров данных** (debug, re-process, data quality). Presentation — для **аналитиков, BI, бизнеса**. Ограничение доступа к staging снижает риск: бизнес не строит отчёты на «сырых» данных, которые могут содержать дубликаты или промежуточные состояния.
