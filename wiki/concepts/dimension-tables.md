# Dimension Tables

**Источник:** [The Data Warehouse Toolkit](../sources/the-data-warehouse-toolkit.md)

## Роль измерения

**Dimension** задаёт **контекст** факта: время, продукт, клиент, география, промо и т.д. В витрине измерение — обычно **широкая** таблица с атрибутами для фильтрации и группировки.

## Surrogate key vs natural key

| | **Surrogate key** | **Natural key** |
|---|-------------------|-----------------|
| Что это | Искусственный ключ (часто `SK`), без бизнес-смысла | Ключ из операционной системы |
| Зачем в DW | Стабильность, поддержка **SCD Type 2**, разные источники с разными форматами | Трассировка к источнику, дедупликация |
| Практика Kimball | **Fact** ссылается на dimension по **surrogate**; natural key хранится как атрибут | |

## Degenerate dimension

Атрибуты, которые логически относятся к измерению, но **не выносятся** в отдельную таблицу — например **номер заказа**, **номер счёта-фактуры** (высокая кардинальность, 1:1 с grain). Хранятся **колонками в fact**.

## Junk dimension

Объединение **множества мелких флагов/кодов** (да/нет, тип доставки, …) в одну искусственную таблицу измерения, чтобы не раздувать fact множеством FK. Уменьшает ширину факта, но нужно следить за комбинаторикой строк.

## Role-playing dimensions

Одно физическое измерение (например **Date**), **несколько FK** из fact: `order_date_key`, `ship_date_key`. Одна таблица дат — **роли** задаются алиасами в запросе/семантическом слое.

## Иерархии

Атрибуты внутри измерения образуют **иерархии** (категория → бренд → SKU; год → квартал → месяц). Для OLAP часто важны **сбалансированные** и **ragged** иерархии (разная глубина ветвей).
