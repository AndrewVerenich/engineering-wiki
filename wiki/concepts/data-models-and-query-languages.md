# Data Models and Query Languages

**Источник:** [Designing Data-Intensive Applications](../sources/designing-data-intensive-applications.md) (гл. 2)

## Суть

Выбор **модели данных** (реляционная, документная, графовая и т.д.) определяет, как выражать сущности, связи и запросы. Нет одной «лучшей» модели: важны паттерны доступа, нормализация vs денормализация, удобство приложения и операций. Языки запросов (SQL, declarative vs imperative) влияют на то, как оптимизатор и движок могут ускорять работу.

## Ключевые модели

| Модель | Сильные стороны | Слабые стороны |
|--------|----------------|----------------|
| **Реляционная** (SQL) | Строгая схема, joins, транзакции, зрелость инструментов | Схема должна быть заранее; сложные иерархии могут быть громоздкими |
| **Документная** (MongoDB, etc.) | Гибкая схема, вложенность, хорошо для «одного агрегата» | Слабые join'ы; денормализация → дублирование → риск рассинхрона |
| **Графовая** (Neo4j, etc.) | Естественна для сильно связанных данных (соцсети, knowledge graph) | Менее зрелые инструменты, сложнее масштабировать |
| **Колоночная / OLAP** | Сканы, агрегации, сжатие; ClickHouse, BigQuery | Плохо для точечных OLTP-запросов, UPDATE дорогой |

## Declarative vs Imperative

- **Declarative** (SQL): описываешь **что** нужно, оптимизатор решает **как** → легче оптимизировать, параллелить.
- **Imperative**: описываешь **шаги** → полный контроль, но оптимизатор не может помочь.

На собеседовании: SQL — declarative, MapReduce — imperative-like (хотя высокоуровневый), Spark — смесь (lazy evaluation + оптимизатор Catalyst).

## Типичные вопросы на интервью

**Q: Когда ты выберешь документную модель вместо реляционной?**
A: Когда данные имеют древовидную/вложенную структуру, схема нестабильна, и связи между документами слабые. Типичный пример: профиль пользователя с вложенными адресами, предпочтениями. Если нужны cross-document joins — это сигнал к реляционной модели.

**Q: Что такое impedance mismatch?**
A: Несоответствие между объектами в коде приложения и их представлением в реляционной БД. ORM частично решает, но не устраняет полностью (N+1, lazy loading проблемы). Документные БД ближе к in-memory структурам, но хуже для связей.

**Q: Почему declarative query languages предпочтительнее?**
A: Оптимизатор может менять план выполнения без изменения запроса: переупорядочивать joins, использовать индексы, параллелить. В imperative подходе любая оптимизация — ручная.

**Q: Что такое schema-on-read vs schema-on-write?**
A: **Schema-on-write** (RDBMS): структура проверяется при записи — строгость, но миграции дорогие. **Schema-on-read** (document, data lake): данные пишутся «как есть», интерпретируются при чтении — гибкость, но ошибки обнаруживаются поздно. В data engineering часто комбинация: raw layer (schema-on-read) → staging/marts (schema-on-write).
