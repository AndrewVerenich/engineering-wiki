# Fact Table Types (Kimball)

**Источник:** [The Data Warehouse Toolkit](../sources/the-data-warehouse-toolkit.md)

Три основных типа факт-таблиц отражают **разные бизнес-процессы** и паттерны загрузки.

## 1. Transaction fact table

- **Событие:** одна строка на **транзакцию/строку** (заказ, оплата, клик).
- **Grain:** максимально детальный уровень процесса.
- **Меры:** обычно **additive** (суммы, количества).
- **Обновления:** часто **append-only**; изменения в OLTP → новые строки или корректировки отдельной логикой.

**Вопрос на собеседовании:** *чем transaction отличается от snapshot?* — transaction описывает **поток событий**, snapshot — **состояние на момент времени**.

## 2. Periodic snapshot fact table

- **Событие:** регулярный **снимок** (например остатки **на конец каждого дня** по складу × SKU).
- **Grain:** фиксированный интервал × сущности (день × магазин × товар).
- **Меры:** часто **semi-additive** (остатки, балансы) — агрегация по времени **не через SUM**, а через «правильный» снимок (end-of-period).

## 3. Accumulating snapshot fact table

- **Событие:** **одна строка на «конвейер»** сущности (заказ, заявка, обращение) с **датами этапов** (создан, оплачен, отгружен).
- Колонки — **вехи процесса**; строка **обновляется** по мере движения по pipeline.
- Удобно для **SLA**, воронок, lead time.

## Factless fact tables

Таблица фактов **без числовых мер** — только ключи (событие «факт произошёл»): участие в промо, назначение врача, факты покрытия. Используются для **счётов событий** и связей many-to-many через bridge (см. [Advanced Dimensional Patterns](advanced-dimensional-patterns.md)).

## Типичные вопросы на интервью

**Q: Назови три типа fact-таблиц и когда каждая уместна.**
A: **Transaction** — одна строка на событие (заказ, клик); append-only, максимальная детализация. **Periodic snapshot** — снимок состояния с фиксированной периодичностью (день × магазин × SKU); для балансов, остатков. **Accumulating snapshot** — одна строка на pipeline/процесс с этапами (вехи: создан → оплачен → отгружен); строка обновляется. Выбор зависит от grain и бизнес-вопроса.

**Q: Что такое factless fact table?**
A: Таблица без числовых мер — только FK на dimensions. Фиксирует **факт события**: участие в промо, назначение врача, покрытие страховки. Используется для подсчёта событий (`COUNT(*)`) и many-to-many связей через bridge.

**Q: Чем transaction fact отличается от periodic snapshot?**
A: Transaction — **поток событий** (каждая продажа, каждый клик); grain — одно событие. Periodic snapshot — **состояние на момент** (остатки на конец дня); grain — интервал × сущность. Transaction facts — additive (суммируй продажи за месяц). Snapshot facts — часто **semi-additive** (баланс нельзя суммировать по дням).

**Q: Когда нужен accumulating snapshot?**
A: Когда бизнес-процесс имеет **этапы с вехами**: заказ создан → оплачен → собран → отгружен → доставлен. Строка обновляется по мере движения. Удобно для: **SLA-мониторинга** (сколько дней от заказа до доставки), воронок, lead time. Минус: требует UPDATE, что в колоночных DWH дороже.
