# Star Schema and Grain

**Источник:** [The Data Warehouse Toolkit](../sources/the-data-warehouse-toolkit.md)

## Star schema (звёзда)

- **Fact table** в центре: **числовые меры** + **внешние ключи** на измерения.
- **Dimension tables** — «лучи»: атрибуты контекста (дата, продукт, клиент, магазин).
- Типично **денормализованные** измерения (мало join’ов, удобно для BI).

## Snowflake (снежинка)

- Измерения **нормализованы** (иерархии вынесены в подтаблицы).
- **Плюсы:** меньше дублирования, проще обновлять части иерархии.
- **Минусы:** больше join’ов, сложнее для пользователей и многих OLAP-движков.
- В Kimball **по умолчанию — звезда**; снежинка — осознанный компромисс.

## Grain (гранулярность)

**Grain** — **одна строка факта = один атом события** на этом уровне детализации.

Примеры формулировок grain:

- «Одна строка = одна позиция строки заказа в день отгрузки»
- «Одна строка = один день × один магазин × один SKU»

**Правило:** grain должен быть **задан явно** до проектирования мер; смена grain — часто **новая факт-таблица** или перезагрузка истории.

## Меры: additivity

| Тип | Пример | Агрегация |
|-----|--------|-----------|
| **Fully additive** | `sales_amount`, `quantity` | SUM по любым измерениям |
| **Semi-additive** | `account_balance`, `inventory_snapshot` | SUM не по всем (часто по времени — «последний снимок») |
| **Non-additive** | `profit_ratio`, средние проценты | Нельзя суммировать; нужны взвешенные формулы на агрегатах |

На собеседовании часто спрашивают: *почему остаток на счёту нельзя суммировать по дням?* — из-за **semi-additive** природы.

## Типичные вопросы на интервью

**Q: Что такое grain и почему его нужно определять первым?**
A: Grain — **что означает одна строка** fact-таблицы. Определяет: какие меры возможны, какие dimensions применимы, какая детализация агрегатов. Если grain размыт (смесь уровней) — метрики будут неправильными. Смена grain — часто **новая** таблица или полная перезагрузка.

**Q: Приведи примеры формулировки grain.**
A: «Одна строка = одна позиция строки заказа». «Одна строка = один день × один магазин × один SKU (ежедневный снимок остатков)». «Одна строка = один клик пользователя на рекламном объявлении».

**Q: Почему star schema, а не snowflake?**
A: Star — **меньше joins** (dimension денормализована), проще для аналитиков и BI-инструментов, лучшая производительность на OLAP-движках. Snowflake — нормализованные dimensions, меньше дублирования, но больше joins и сложнее для пользователей. Kimball рекомендует star по умолчанию; snowflake — осознанный компромисс для конкретных случаев.

**Q: Что такое semi-additive мера и почему это важно?**
A: Мера, которую нельзя суммировать по **всем** измерениям. Пример: `account_balance` — можно суммировать по клиентам (сколько всего на счетах), но **нельзя** суммировать по дням (это не «общий баланс за три дня» — нужен последний snapshot). На собеседовании: «почему остаток на счёте нельзя суммировать по дням?»
