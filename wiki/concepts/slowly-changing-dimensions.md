# Slowly Changing Dimensions (SCD)

**Источник:** [The Data Warehouse Toolkit](../sources/the-data-warehouse-toolkit.md)

**SCD** — изменение атрибутов измерения во времени (смена адреса клиента, переназначение категории товара). Выбор типа — баланс между **историей**, **простотой** и **согласованностью** отчётов.

## Type 1: перезапись

- Старое значение **заменяется** новым.
- **История не сохраняется**; отчёты за прошлое «как будто всегда было новое значение».
- **Когда:** исправление ошибок, неважна история.

## Type 2: новая строка (история)

- При изменении — **новая строка** в dimension с новым **surrogate key**; старая остаётся.
- Обычно колонки **valid_from / valid_to** или **is_current**, **version**.
- **Fact** ссылается на тот surrogate, который был актуален на момент события — так сохраняется **историческая правда**.
- **Когда:** нужна корректная история (типично для клиента, продукта).

## Type 3: «альтернативное» поле

- Добавляется колонка **previous_*** или «текущая vs прошлая» категория — **очень ограниченная** история (часто одно сравнение).
- Редко в чистом виде; чаще как компромисс.

## Гибриды и расширения

- **Type 6** (в терминологии Kimball): комбинация Type 1 + 2 + 3 — текущие атрибуты дублируются для удобства отчётов «как сейчас» и истории.
- Несколько **медленно меняющихся** атрибутов в одном измерении могут требовать **разных** стратегий по столбцам.

## Типичные вопросы на интервью

1. *Когда Type 1 vs Type 2?* — нужна ли **историческая достоверность** при срезах по времени.
2. *Как fact связывается с Type 2?* — по **surrogate key на дату факта** (SCD даёт правильную версию измерения).
3. *Что такое **late-arriving dimensions**?* — факт пришёл раньше, чем строка измерения; нужны политики default/late binding (продвинутый кейс).
