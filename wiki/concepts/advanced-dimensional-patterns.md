# Advanced Dimensional Patterns

**Источник:** [The Data Warehouse Toolkit](../sources/the-data-warehouse-toolkit.md)

Паттерны для **многие-ко-многим**, разреженности атрибутов и производительности.

## Bridge table (мост)

Когда связь **many-to-many** между фактом и измерением (пациент ↔ диагнозы, счёт ↔ несколько ответственных), вводится **bridge** с весами/приоритетами. Усложняет **additivity** — двойной счёт фактов, если не осторожничать с распределением.

## Factless fact table

См. [Fact Table Types (Kimball)](fact-table-types-kimball.md): только ключи; используется для событий покрытия, участия, назначений.

## Minidimension

Вынесение **тяжёлых или быстро меняющихся** атрибутов (много полей) в отдельное маленькое измерение, на которое fact ссылается дополнительным FK — чтобы не раздувать основной **SCD Type 2** (меньше версий основной строки).

## Outrigger

**Редкий** приём: FK из **dimension** на другую таблицу (мини-нормализация внутри звезды). Kimball обычно **остерегает** чрезмерных outrigger’ов — риск «снежинки» и сложности для пользователей.

## Snowflake умеренно

Глубокая нормализация измерений только там, где **реально** shared sub-иерархии и объём данных; иначе — **star**.

## Типичные вопросы на интервью

**Q: Как моделировать many-to-many связь между фактом и измерением?**
A: Через **bridge table** — промежуточная таблица с FK на fact (или grain) и dimension, плюс вес/приоритет. Пример: пациент с несколькими диагнозами. Нужно следить за **двойным счётом**: если вес не распределён, SUM(amount) будет завышена. Решение: allocation factor в bridge.

**Q: Что такое minidimension и когда она нужна?**
A: Вынесение **быстро меняющихся** атрибутов в отдельное маленькое измерение. Пример: demographic band клиента (возрастная группа, income band) — меняется чаще, чем основной профиль. Без minidimension — SCD Type 2 на основной dimension создаст **слишком много версий**. С minidimension — основная dimension стабильна, а быстрые изменения живут отдельно.

**Q: Когда snowflake оправдана?**
A: Когда shared sub-иерархия используется **несколькими dimensions** (например geography: country → region → city), и объём данных в подтаблице существенный. Но Kimball рекомендует star по умолчанию и snowflake — только при **осознанной необходимости**.
