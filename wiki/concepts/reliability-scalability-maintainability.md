# Reliability, Scalability, Maintainability

**Источник:** [Designing Data-Intensive Applications](../sources/designing-data-intensive-applications.md) (гл. 1)

## Суть

Три свойства, которыми описывают «хорошие» data-intensive приложения:

- **Reliability** — система продолжает корректно работать при сбоях (железо, ПО, человек).
- **Scalability** — справляться с ростом нагрузки разумными средствами (масштабирование **up** или **out**, узкие места).
- **Maintainability** — операции, простота эволюции: наблюдаемость, управляемость, простота для новых людей.

## Ключевые термины

| Термин | Определение |
|--------|-------------|
| **Fault vs failure** | Fault — отклонение компонента; failure — сбой всей системы. Цель — **fault-tolerant**: система работает, даже если часть компонентов сбоит. |
| **Latency vs response time** | Latency — время ожидания обработки; response time — полное время от запроса до ответа (включая сеть). Для SLA важны **перцентили** (p50, p95, p99), а не среднее. |
| **Tail latency** | Высокие перцентили (p99, p999): малая доля запросов, но часто затрагивает самых ценных пользователей (много данных). |
| **Vertical vs horizontal scaling** | Scale up (мощнее машина) vs scale out (больше машин). Большинство data-intensive систем — горизонтальное масштабирование. |
| **Elastic scaling** | Автоматическое добавление/снятие ресурсов при изменении нагрузки (облачный паттерн). |
| **Operability** | Наблюдаемость, мониторинг, автоматизация рутины, документация — часть maintainability. |

## Типичные вопросы на интервью

**Q: Как бы ты описал три свойства хорошей data-intensive системы?**
A: **Reliability** — корректная работа при сбоях (hardware, software, human error). **Scalability** — способность справляться с ростом нагрузки путём вертикального или горизонтального масштабирования; оценивается через load parameters и перцентили latency. **Maintainability** — простота эксплуатации, изменений и понимания новыми людьми.

**Q: Почему среднее время ответа — плохая метрика для SLA?**
A: Среднее скрывает outlier'ы. Пользователи с p99/p999 latency — часто самые активные (больше данных). Поэтому SLA задают через перцентили: p50 для «типичного» опыта, p95/p99 для «хвоста».

**Q: В чём разница между fault tolerance и disaster recovery?**
A: Fault tolerance — система **продолжает работать** при отдельных сбоях (реплики, retry, failover). Disaster recovery — **восстановление** после катастрофы (бэкапы, failover в другой регион). Первое — проактивно, второе — реактивно.

**Q: Как ты решаешь, масштабировать вертикально или горизонтально?**
A: Вертикально — пока хватает одной машины и это проще (меньше сложности распределёнки). Горизонтально — когда упираемся в потолок одной машины, нужна отказоустойчивость, или нагрузка непредсказуема (elastic). В data engineering горизонтальное масштабирование — обычно норма (Kafka, Spark, ClickHouse cluster).
